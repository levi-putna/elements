# Instant Strata Elements — Agent Guide

This repository is the **UI element library for Instant Strata**, a strata (property)
management platform. Every component the project uses lives here as owned source code.
This file is the entry point for AI agents building Instant Strata UI.

> **Scope:** Use only the elements in this library. Do not introduce other component
> libraries or invent new design tokens. If something is missing, compose it from the
> existing primitives.

## Stack & conventions

- **shadcn/ui** distribution model (base-nova style): components are copied into the
  consumer's `components/ui/` via the shadcn CLI — there is no installable package.
- Underlying primitives are **Base UI** (`@base-ui/react`), **not** Radix.
- Styling is **Tailwind CSS** with CSS variables; class merging via `cn()` from `@/lib/utils`.
- Each element lives in `registry/<name>/`:
  - `<name>.tsx` — component source (the source of truth for behaviour)
  - `registry.json` — shadcn install manifest
  - `<name>.md` — **agent documentation** (the source of truth for *when/why/how*)

## How to use this guide

1. Read the **non-negotiable rules** below and [DESIGN.md](DESIGN.md) for brand/tokens.
2. Find the right element in the **index** and open its `registry/<name>/<name>.md`.
3. Each `.md` has frontmatter (category, keywords, install, related) plus
   *When to use / When NOT to use / Anatomy / API / Variants / Do-Don't*.
4. Prefer composing existing elements over writing new ones.

## Non-negotiable rules

- **Radius:** 4px on buttons/inputs, 8px on cards. **No pill shapes.**
- **Colour:** White is the dominant background; off-white (`#EEF2E3`) for subtle
  alternation. Forest green (`#043F2E`) = primary actions + dark sections + text.
  Lime (`#C8F169`) = accent/CTA, used sparingly. Never white text on lime.
- **Type:** Young Serif (weight 400) for display/headings; Inter for UI/body/labels.
- **Dark sections** (forest) used sparingly — max 3 per page (hero, one testimonial, footer).
- **Buttons:** max two per CTA group; one primary.

## Conventions for these docs

- Each `<name>.md` is the **source of truth** for description, categorisation, and
  when-to-use. The one-line `registry.json` description and the docs-site page should
  be derived from it, not hand-maintained separately.
- A machine-readable index (`llms.txt` / `index.json`) can be generated from the
  frontmatter across all `<name>.md` files.
